## Detected Conflicts / Design Flaws

### 1. Report Target Type for Question Reports
- **Issue**: In `7_review_and_report`, reporting a question during an attempt targets the `quiz_question` snapshot. However, the existing `report` table only allows `target_type` values `'quiz'`, `'question'`, `'comment'`. Adding `'quiz_question'` requires altering the check constraint. Alternatively, we could overload `'question'` to mean both original and snapshot, but that would require a separate column to disambiguate. The design’s proposal to use `'quiz_question'` is cleaner but needs DB migration.
- **Recommendation**: Add `'quiz_question'` to the check constraint and include `attempt_id` column (already in design) to provide context.

### 2. LLM Global Cache – User ID Nullability
- **Issue**: The `llm_log` table currently has `user_id` as a foreign key with `ON DELETE SET NULL`. Making `user_id` nullable for global cache is fine, but existing queries (e.g., in `ILlmLogRepository.FindCachedAsync`) may assume `user_id` is not null. They must be updated to include `OR user_id IS NULL`.
- **Risk**: If any queries inadvertently filter out nulls, global cache entries won’t be used. Need thorough review.
- **Recommendation**: Add the unique index as suggested, and update repository methods to include null in the lookup. Also consider adding a `is_global` flag for clarity.

### 3. Note Updates Affect Multiple Quizzes
- **Issue**: In `10_quiz_note_source`, updating a note will change its content for **all** quizzes that reference it. This is probably intended, but the user should be aware. No conflict, just a design point to confirm.

### 4. Source Viewing – Modal vs Navigation
- **Issue**: The design offers both a modal preview and navigation to source detail. This could lead to inconsistent UX. Choose one approach (modal for quick glance, navigation for full reading). The user should decide.

### 5. Search “All” Tab Pagination
- **Issue**: The `15_search_pagination` design chooses to keep “All” as a limited preview (20 results) to avoid complex union queries. This may confuse users expecting full pagination. If later we decide to paginate “All”, it will require significant backend work. We should confirm this trade‑off is acceptable.

### 6. TTS Cache Storage
- **Issue**: The TTS cache design uses file storage. If many words are pre‑generated, the filesystem may become cluttered. A cleanup policy (e.g., LRU) should be considered but is not in the initial design. Also, file permissions and serving via the gateway need to be secure (no path traversal). The design mentions using a hash‑based filename, which is safe.

### 7. Avatar File Storage Service vs Source Parsing Service
- **Issue**: Two separate microservices are proposed for file handling (parsing and avatar storage). They could be combined into one generic file service, but the requirements differ (parsing vs storage). Combining might overcomplicate. It’s acceptable to keep them separate for now.

### 8. Email Rate Limiting
- **Issue**: The `RateLimitingMiddleware` currently only applies to login. The `8_reset_password` design suggests rate limiting on forgot‑password endpoints. We need to extend the middleware or create a new one. Ensure Redis is used for distributed counting.

### 9. Admin Undo Report – Logging
- **Issue**: The `2_admin_report_solved_view` design proposes logging undo actions in `admin_log`. Good, but ensure the log entry contains enough detail (e.g., previous resolution info). Not critical.

### 10. API Metrics Storage
- **Issue**: The `21_api_log` design suggests using Redis for high‑frequency metrics and PostgreSQL for aggregates. If Redis is not persisted, metrics may be lost on restart. For a small VPS, this might be acceptable. Decide if we need historical metrics.

### 11. Breadcrumb Truncation on Mobile
- **Issue**: The breadcrumb design uses a `maxItems` prop with ellipsis. On very small screens, even two items may overflow. We may need to rely on Ant Design’s built‑in responsiveness or add CSS overflow handling. Not a major flaw.

### 12. i18n Effort
- **Issue**: Implementing language switching (`17_setting`) requires extracting all UI strings into translation files. This is a large task. Ensure it’s properly scoped and prioritize key pages first.

### 13. Word of the Day – Fallback List
- **Issue**: The fallback word list should be stored somewhere (e.g., app settings or a static JSON). If the list is large, it could be a configuration table. For simplicity, a hardcoded array in the backend is acceptable.

### 14. Concurrency in Quiz Attempts
- **Issue**: The design doc mentions “last write wins” for simultaneous submissions. This is acceptable, but we should ensure that answers are not overwritten incorrectly. The existing attempt answer storage likely handles this by updating the same answer row. No conflict.

### 15. Tag Selector – Performance
- **Issue**: Fetching all tags on component mount may become slow if there are thousands of tags. Caching in the store (as designed) mitigates this. But we should also consider adding a search/filter in the dropdown if tag count grows large.

These questions and conflicts should be resolved before coding to ensure smooth implementation and alignment with overall architecture.
